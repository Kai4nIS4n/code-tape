# GitNexus + OpenViking 静态知识清单契约

本文说明 code-tape 的双层契约：本地 agent 先用 GitNexus 建立代码图谱并做影响分析，PR 再由 CI 阻断兜底；OpenViking 在本仓库中只作为静态权威知识清单，不要求组员连接任何服务。

## 本地 agent 工作流

所有 agent 在改代码前先运行：

```bash
npm run agent:bootstrap
```

每次开始任务和提交前运行：

```bash
npm run contract:local
```

`contract:local` 会强制执行 `npx --yes gitnexus@1.6.5 analyze --force --index-only`，只刷新 `.gitnexus/` 索引，不注入或重写 `AGENTS.md` / `CLAUDE.md`。随后脚本检查当前 diff 是否触碰关键骨架。命中以下区域时，必须阅读脚本输出的 GitNexus 建议，并在 PR 的 “GitNexus 影响分析摘要” 中写明结论：

- `apps/web/src/shared/recording-schema/`
- `apps/web/src/features/runtime-preview/`
- `apps/web/src/features/library/recordingStore*`
- `apps/web/src/features/player/replay*`、`packageLoader*`、`timelineClock*`、`mediaClockAdapter*`
- `.github/workflows/`、`scripts/workflows/`
- `README.md`、`AGENTS.md`、`CLAUDE.md`、PRD、技术方案、流程文档

本仓库不自动安装 git hooks，避免改动组员个人环境；契约通过 agent 指令、npm scripts、PR 模板和 CI 共同生效。

## PR / CI 契约

PR 阶段会运行：

```bash
npm run contract:gitnexus
npm run contract:openviking:check
```

`contract:gitnexus` 同样强制刷新 GitNexus 索引。关键骨架变更必须同时满足：

- 有匹配的契约测试或 workflow 测试变更。
- PR 模板中填写了 GitNexus 影响分析摘要。

`contract:openviking:check` 只校验 `docs/contracts/openviking.resources.json`，不会连接 OpenViking 服务，也不会上传 PR 内容。

## OpenViking 静态清单

OpenViking 在本仓库中的用途是固定“哪些文档属于 agent 必读的权威上下文”。组员和 CI 都不需要配置服务地址、密钥、账号或本机同步任务。

`docs/contracts/openviking.resources.json` 是静态清单入口，包含：

- `rootUri=/projects/code-tape`
- 每个权威资源的 `path`
- 对应的逻辑 `uri`
- 当前文件内容的 `sha256`
- 该资源进入清单的 `reason`

agent 需要上下文时直接读取仓库中的清单和对应本地文件，不通过远程服务获取项目知识。

## 资源清单维护

`docs/contracts/openviking.resources.json` 是 OpenViking 项目知识库入口。任何权威资源内容变化后，都必须更新对应 `sha256`。

更新步骤：

1. 修改文档或规则。
2. 重新计算变更文件的 sha256。
3. 更新 `docs/contracts/openviking.resources.json`。
4. 运行 `npm run contract:openviking:check`。

PR 阶段只校验清单，不做远程同步；合入 main 后也不会触发 OpenViking 服务写入。
