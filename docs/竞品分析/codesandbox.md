## 1. 对应模块
本调研主要服务于 code-tape 项目中的以下模块：

- **回放与录制协同 (P0/P1)**：参考 IDE 界面中“代码-终端-预览”的布局分配逻辑以及协作光标。

## 2. 结论摘要
CodeSandbox 的核心竞争力在于**开箱即用的前端云环境**与**极低的分享协作阻力**。它的演进证明了“在浏览器内通过 IFRAME + ServiceWorker”完成前端沙箱运行不仅完全可行，而且是针对教学、演示场景体验最佳的方案。
对本项目的启发：P0 阶段的展示沙箱应**坚决走纯前端 iframe 路线**，避免过早引入后端容器；在分享层，应做到 URL 及拿即用，并支持通过 `<iframe src="..." />` 形式嵌入第三方文档（如 Notion/飞书），从而扩大产品病毒式传播能力。

## 4. 产品定位、目标用户和核心路径

### 产品定位
CodeSandbox 定位为“云端 Web 开发环境”。它将 IDE 的体验搬到浏览器中，解决了本地环境配置繁琐、代码片段难以快速分享预览的痛点。

### 目标用户
- **前端工程师 / 开源作者**：制作快速的可交互 Demo 放在 GitHub Readme 或官方文档中。
- **教学与讲师**：发放带有预设模板和错误代码的 Sandbox 链接供学生练习修复。
- **面试官**：用于远程结对编程（Live Collaboration）与技术面考察。

### 核心使用路径
1. 选择预置环境（HTML/JS, React, Vue, Node 等）初始化 Sandbox。
2. 浏览器加载 Monaco Editor 和预览视图，代码键入后后台（或纯前端）即刻触发构建。
3. 如果是前端沙箱，打包操作在客户端完成，实时渲染在旁侧的跨域 iframe 中。
4. 复制浏览器地址栏的 Hash URL 即可分享；或点击 Share 获取嵌入代码（Embed Code）。

## 6. 代码运行与展示方式深度分析

CodeSandbox 目前维护了两套沙箱体系，这对我们有极强的参考意义：

### 1. Browser Sandbox (浏览器纯前端沙箱)
这是他们起家的能力（开源版本为 Sandpack）。当用户选择如 React/Vanilla 模板时：
- 不会分配真实服务器。
- CodeSandbox 在浏览器里内嵌了一个基于 webpack API 的微型打包器。
- 代码变更后，在本地浏览器计算 AST，拦截 npm 请求。
- 把最终 JS 产物扔给一个跨域的 iframe 执行。
- **优点**：毫秒级启动，免服务器成本费，断网（如果有缓存）甚至短期内可用。

### 2. Cloud Sandbox (MicroVMs 微虚拟机)
针对 Node.js、Next.js 等需要后端能力的框架：
- 利用 AWS Firecracker 在极短时间内（约 1 秒）克隆一台微型 Linux 虚拟机。
- 并将终端流推给前端。

**👉 结论推导：我们不做 Cloud Sandbox，坚定走 Browser Sandbox 路线。**

## 7. 分享、存储、权限与协作

- **极低门槛分享**：用户一旦生成代码或保存录制，URL Hash 就可直接发送。接收者打开时，右侧会自动呈现运行结果。
- **Fork 机制**：别人分享的代码是只读的，如果你试图编辑，系统会毫无感知地为你 Fork 生成一份副本（并更新 URL）。这个非常适合 code-tape 教学场景中“学生打断录播尝试修改”的动作。
- **嵌入 (Embed) 体验**：提供 `?view=preview` 等参数，可以轻易将演示嵌入到博客、飞书文档中。

## 8. 对 P0 选择 iframe 前端展示路线的利弊判断

在 P0 阶段，本项目决定采用 iframe（结合 srcdoc 甚至 Web Worker 注入）而非云端 Docker/VM，利弊判断如下：

| 维度 | iframe 纯前端沙箱 (P0 决策) | Docker/VM 云端沙箱 (竞品对比) |
| --- | --- | --- |
| **开发成本** | **极低**：几十行隔离代码即可出图 | 高：需处理调度、K8s池、保活等 |
| **响应速度** | **极快**：无需长连接，加载完录制包即运行 | 慢：需等待容器分配，存在冷启动时间 |
| **服务器成本**| **零**：纯吃用户所在客户端算力 | 极高：每个回放/录制者单独配实例 |
| **安全性** | **可控，切中考核点**：利用 `sandbox="allow-scripts"` 严格限制 API，能够隔绝绝大部分安全影响并**实现禁止弹窗（契合 P1 挑战项）** | 难控：容易遭到挖矿脚本/溢出攻击 |
| **功能支持面**| **局限**：仅能支持无后端依赖的前端 JS/HTML，但**可实现 Console 拦截与输出（契合 P1 进阶实现）** | 广：支持全栈，终端，真实文件系统 |

**结论**：在当前 P0（1个月）有限周期内，优先保障“代码可回放 + 结果可视”，局限性是可以接受的代价。坚决执行 iframe 方案。

## 9. 对分享/存储体验的 2 条具体建议

1. **“无损 Fork”与观看者保护隔离**：
   参考 CodeSandbox，当观众在观看代码讲解时，如果暂停并试图编辑代码，录制播放器的事件轨道**不能被污染**。必须在底层内存中生成一个针对当前快照的 Fork，所有用户的新代码变更作用于此 Fork 在 iframe 渲染；点击“继续播放”时，回滚回原始讲师轨道的状态。
2. **沉淀标准 Embed iframe SDK**：
   在 P1 阶段设计回放链接时，考虑输出类似于 `https://our-project/embed/:recordId?theme=dark&hideEditor=true` 的 URL。这能让本项目的产物直接作为“动态可互动的代码块”嵌入他人博客。
