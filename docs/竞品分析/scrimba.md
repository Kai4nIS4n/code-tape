## 4. 产品定位、目标用户和核心路径

### 4.1 产品定位

Scrimba 是面向编程学习的互动式课程平台，核心形态是 interactive coding screencast。它不是只让用户观看视频，而是把讲师讲解、代码编辑器、运行环境和练习任务融合成同一个学习播放器。

官方文章将 Scrimba 的差异点描述为：Scrimba 记录的是浏览器事件而不是视频像素，因此学习者可以暂停课程、修改讲师代码并立即看到运行结果。这说明 Scrimba 的核心价值不是“录屏清晰度”，而是“讲解过程可被接管和再执行”。

### 4.2 目标用户

从 Scrimba 首页和官方文章看，主要目标用户包括：

- 前端入门学习者：学习 HTML、CSS、JavaScript、React 等 Web 技术。
- 转行或求职用户：通过 Frontend Developer Career Path 等路径完成体系化训练。
- 需要大量动手练习的学习者：通过 challenge、项目课和 AI feedback 强化实践。
- 教学内容消费者：跟随讲师步骤学习，同时能在关键时刻暂停试写代码。

### 4.3 核心使用路径

典型路径可以拆成：

1. 进入首页或课程目录，选择 career path、course 或 lesson。
2. 进入 scrim 播放器，观看讲师讲解和代码演示。
3. 播放过程中查看代码变化、运行结果和预览。
4. 在需要理解或练习时暂停播放器。
5. 接管编辑器，修改讲师当前代码。
6. 运行代码并查看 preview、console、测试或 feedback。
7. 继续播放时，用户修改会被讲师原始事件轨道覆盖，回到讲师原始讲解路径。
8. 完成 challenge 或课程节点，进度被记录到课程组织结构中。

对 code-tape 来说，这条路径最重要的启发是：播放器不是被动视频容器，而是一个可以被观看者临时接管的代码工作台。

## 6. 录制数据范围分析

Scrimba 没有公开完整录制数据 schema，因此下面按“可确认 / 可推断 / 未确认”拆分。

### 6.1 可从官方资料确认

- Scrimba 的 scrim 不是传统视频像素录制，而是基于浏览器事件的互动式内容。
- 播放器和代码编辑器高度融合，观看者可以暂停并编辑代码。
- 学习者修改代码后，可以即时看到运行结果或反馈。

### 6.2 人工观察可确认

- 播放过程中可以看到讲师光标、选区、鼠标轨迹和快捷键提示。
- 暂停后用户可以修改代码，但继续播放时用户修改会被讲师原始轨道覆盖。
- 运行结果区域包含页面预览、console 和测试结果。
- 支持保存、fork 和分享某个 scrim 或某个时间点。

### 6.3 基于产品形态的合理推断

为了支持上述回放体验，Scrimba 至少需要记录或重建：

- 代码内容随时间的变化。
- 播放时间轴与代码状态之间的映射。
- 光标、选区、鼠标轨迹和快捷键提示。
- 当前 lesson 或 challenge 的初始文件状态。
- 运行预览所需的 HTML/CSS/JS 或项目上下文。
- 课程节点、挑战节点和进度信息。

### 6.4 未确认，不建议直接作为事实写入 PRD

- 是否记录讲师摄像头。
- 音频是独立音轨、浏览器事件轨道的一部分，还是经过其他格式封装。
- seek 的状态恢复是基于快照、事件重放，还是混合策略。

对 code-tape 的影响：Scrimba 可以作为产品路径参考，但不能作为底层事件 schema 的直接依据。code-tape 仍需要自行定义 `content-change`、`selection-change`、`mouse-move`、`shortcut`、`record-marker` 等事件。

## 8. 分享、存储、权限与课程组织

### 8.1 课程组织

Scrimba 以课程、职业路径、lesson、challenge 组织学习内容。对学习者而言，scrim 是一个课程节点；对平台而言，多个 scrim 组成课程或职业路径。

可参考的信息架构：

- Path：体系化学习路径，例如前端职业路径。
- Course：围绕某个技术主题的课程。
- Lesson / Scrim：单个互动式讲解单元。
- Challenge：要求学习者主动完成代码任务的练习单元。
- Progress：课程完成度或学习进度。

### 8.2 分享与存储

官方公开资料能确认 Scrimba 是云端课程平台，人工观察补充如下：

- 支持保存。
- 支持 fork 某个 scrim。
- 支持分享某个 scrim。
- 支持分享某个时间点。
- 无需登录即可进入完整播放器并保存学习进度。

对 code-tape 来说，分享某个时间点尤其值得参考：录制包不只需要整体播放链接，还可以在 P1 设计 `?t=12345` 这样的时间点定位能力。

### 8.3 对 code-tape 的取舍

code-tape P0 不应直接做完整课程平台。更合适的路径是：

- P0：生成可本地保存或上传的单个录制包，支持回放。
- P1：增加回放列表、重命名、删除、在线播放。
- P1+：再考虑课程目录、章节、challenge、分享链接和权限。

## 9. 核心结论

1. Scrimba 值得作为 code-tape P0 的重点产品参考对象，尤其是“播放器即 IDE”的互动式回放模式。
2. Scrimba 对底层录制 schema 的参考价值有限，因为其浏览器事件格式和播放器实现并未公开。
3. code-tape P0 应借鉴 Scrimba 的产品路径，但实现上仍应围绕本项目 PRD 的事件流：内容变更、光标选区、鼠标轨迹、快捷键、音视频轨道。
4. 暂停后编辑和运行代码是 Scrimba 最有辨识度的体验；其“继续播放覆盖用户修改”的规则很适合 code-tape 借鉴，用于避免观看者临时编辑污染讲师原始事件流。
5. Scrimba 的 challenge、AI feedback、课程路径和学习进度适合 P1/P1+，不建议进入 P0 范围。
6. Scrimba 播放中可展示光标、选区、鼠标轨迹和快捷键提示，这与 code-tape PRD 的回放展示要求高度一致，应作为 P0 回放层的重要参考。

## 11. 对 P0 模块的启发

### 11.1 代码编辑器模块

- 编辑器应是录制和回放的主舞台，而不是视频旁边的附属面板。
- P0 至少支持 JS/TS 高亮、基础编辑、语言信息和运行入口。
- 录制包中建议预留 `files` 或 `language` 字段，避免后续从单文件扩展到多文件时返工。
- Scrimba 支持补全、格式化、注释快捷键和文件新增/删除；code-tape P0 可优先借助成熟编辑器内置能力，不需要自研。

### 11.2 操作录制模块

- Scrimba 证明“记录操作事件而非录屏”是正确方向。
- code-tape 应明确事件轨道，例如 `content-change`、`selection-change`、`mouse-move`、`shortcut`。
- 为暂停和继续录制设计统一时间轴，避免回放时事件与媒体不同步。
- Scrimba 播放中可见光标、选区、鼠标轨迹和快捷键提示，说明这些操作层信息对理解讲解过程有明显价值，应进入 P0 录制范围。

### 11.3 回放模块

- 回放页面应优先保证代码状态和讲解时间轴同步。
- seek 是 P0 风险点，应尽早做 PoC，至少实现“恢复目标时间点最近快照/最近状态”。
- 建议区分原始讲师轨道和观看者临时编辑轨道，为未来“暂停后接管编辑”留扩展口。
- 倍速档位可先覆盖 PRD 要求，再参考 Scrimba 增加更细的速度选项。
- 若后续支持观看者暂停编辑，建议采用 Scrimba 类似策略：继续播放时回到讲师原始轨道，用户修改通过 fork/save 另存。

### 11.4 代码执行/展示沙箱

- Scrimba 的即时运行体验很适合作为 code-tape 的演示亮点。
- P0 推荐优先选择前端 iframe sandbox 展示路线，成本低于后端安全沙箱。
- Console 捕获和测试结果展示有明确参考价值，但可以作为 P0 可选增强或 P1 完善。
- AI feedback 在官方资料中属于 Scrimba challenge 能力，本次播放器观察未看到 AI 工具，不建议进入 P0。

## 12. 推荐方案

Scrimba 值得作为 P0 重点参考对象，但参考重点应限定在产品路径和交互体验，而不是试图复刻完整平台。

推荐 code-tape P0 采用以下取舍：

1. 采用事件流录制与回放，而不是视频录屏。
2. 回放编辑器首版以“还原讲师代码状态”为主，先保证播放、暂停、倍速、seek 正确。
3. 代码运行采用前端 iframe sandbox 或轻量预览方案，优先满足演示。
4. 回放层必须展示光标、选区、鼠标轨迹和快捷键提示。
5. 暂停后编辑可以作为 P1 能力，但 P0 录制包应预留观看者临时编辑轨道。
6. 录制包中预留观看者接管、字幕、章节、多文件和语言扩展字段。
7. challenge、AI feedback、课程路径、学习进度不进入 P0。

## 13. 风险与待验证点

### 13.1 产品风险

- 如果 P0 过早加入观看者暂停后编辑，可能影响原始回放状态恢复。
- 如果 UI 过度模仿课程平台，会稀释“录制与回放工具”的主目标。
- 如果代码运行直接执行用户 JS，可能污染宿主页面或产生安全问题。

### 13.2 技术风险

- seek 后状态恢复复杂，需要快照或索引策略。
- 媒体轨道与事件轨道可能漂移，需要统一相对时间。
- 多文件、运行预览、console 捕获会扩大录制包结构。
- 用户接管编辑需要区分原始事件轨和用户编辑轨，否则继续播放语义不清。

### 13.3 仍需后续验证的问题

本次调研已经补齐播放器核心交互，但底层实现仍无法从公开资料确认：

1. Scrimba 的 seek 状态恢复是基于完整事件重放、状态快照，还是二者混合。
2. 音频与事件流如何封装和对齐。
3. 讲师摄像头是否作为常规录制轨道存在。
4. 保存学习进度是否完全本地化，还是存在未登录用户的临时云端身份。
5. fork 后的用户修改如何存储、是否可以继续分享和二次编辑。
